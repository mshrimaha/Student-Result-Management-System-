1. Overview

This application is a lightweight Student Result Management System built with the Flask web framework. It allows administrators to maintain student records (name, ID and marks across five subjects) and lets students publicly look up their own results using their Student ID and full name as a privacy check. Results can be downloaded as a professionally formatted PDF marks card.

The two primary user flows are:

  • Public lookup — anyone can visit /results, identify themselves and view or download their marks.
  • Admin management — an authenticated administrator can add, edit and delete student records from /admin.

2. Technologies Used

Python 3.11

Modern, well-supported Python runtime — chosen for readability and a rich ecosystem of web and PDF libraries.

Flask

Minimal Python web framework that provides routing, request handling, sessions and Jinja2 templating without the overhead of a full-stack framework.

Jinja2 Templates

HTML is rendered server-side with Jinja2, with a shared base.html layout extended by every page for a consistent header, footer and flash-message system.

Replit Database

A managed key-value store accessed via the official replit Python package. Each student is stored under a student:<ID> key as a JSON blob — no schema migrations required.

ReportLab

Pure-Python PDF generation library used to build the official-looking marks card with tables, branded colors, signature lines and metadata.

Vanilla HTML & CSS

No frontend framework — a single style.css defines the design tokens (colors, spacing, radii) and components for a clean, responsive UI.

4. Data Model

Each student record is stored as a JSON document under the key student:<UPPERCASE_ID>:

{
  "id":   "S001",
  "name": "Jane Doe",
  "marks": {
    "Mathematics": 92,
    "Science":     88,
    "English":     76,
    "History":     81,
    "Computer":    95
  }
}

The five subjects (Mathematics, Science, English, History, Computer) are defined once as a constant in main.py, so the admin form, results page and PDF all stay in sync if the list ever changes.

5. Grading System Logic

Whenever a result is displayed (on screen or in the PDF), the system computes the totals and grade on the fly from the stored marks. There is no need to recompute and re-store these values when a record is edited.

5.1 Totals & Average

  • Total = sum of marks across all five subjects.
  • Maximum = number of subjects × 100 = 500.
  • Average (%) = total ÷ number of subjects, rounded to two decimal places.

5.2 Grade Bands

The final grade is derived from the average:

Average (%)GradeDescription
90 – 100A+Outstanding
80 – 89AExcellent
70 – 79BVery Good
60 – 69CGood
50 – 59DSatisfactory
Below 50FFail

5.3 Pass / Fail Status

A student is marked PASS only if they score at least 35 marks out of 100 in every subject. Failing in even one subject results in an overall FAIL, regardless of the average. This is a simple safeguard against a high average masking a single very poor performance.

5.4 Input Validation

The admin form enforces these rules before saving a record:

  • Student ID and Name are required and trimmed of whitespace; IDs are normalized to uppercase.
  • Marks for every subject are required and must be whole numbers between 0 and 100.
  • Adding a record with an existing ID is rejected; editing a record to use an ID that already belongs to another student is also rejected.

6. Security Features

6.1 Admin Authentication

The /admin dashboard and all add / edit / delete routes are protected by a @login_required decorator. Unauthenticated visitors are redirected to /login. Credentials are validated against the ADMIN_USERNAME and ADMIN_PASSWORD environment variables (defaulting to admin / admin123 for first-run convenience — these should be changed for any real deployment).

6.2 Server-Side Sessions

Flask's signed-cookie sessions are used to track the logged-in admin and to remember the student who just successfully looked up a result (so they — and only they — can download the matching PDF). The signing key comes from the SESSION_SECRET environment variable, which is never stored in source code.

6.3 Two-Factor Result Lookup

On the public results page, a student must enter both their Student ID and their full registered name to retrieve a result. Even if someone knows or guesses an ID, they cannot view the marks without also knowing the name on file. Error messages are intentionally generic ("we could not find a result matching that Student ID and name") so the form does not reveal whether a given ID exists.

6.4 Math Captcha

Each visit to the results page generates a fresh arithmetic challenge (e.g. "What is 7 + 4?") and stores the expected answer in the user's session. The form will only accept a result lookup when the captcha answer matches. After each submission the captcha is consumed and regenerated — a stolen or replayed answer cannot be reused. This blocks naïve scrapers and bots from enumerating IDs.

6.5 PDF Download Authorization

The PDF route /results/pdf does not accept a Student ID parameter. Instead it reads the ID and name from the session token issued only after a successful captcha-verified lookup, then re-validates that the stored record still matches. This prevents direct-link scraping of arbitrary students' PDFs.

6.6 Other Hardening

  • Destructive admin actions (delete student) only respond to POST requests, with a JavaScript confirmation prompt.
  • All user input is escaped automatically by Jinja2's default autoescaping when rendered in HTML.
  • The Replit Database is accessed only through helper functions — there is no raw query string anywhere in the app.

7. PDF Marks Card

After a successful lookup, the student can click Download as PDF to receive an official-looking marks card built with ReportLab. The layout includes:

  • An institution header with name and office tagline (configurable).
  • Student name, ID, date of issue and examination label.
  • A numbered subject-wise marks table showing maximum marks, marks obtained, and per-subject pass/fail.
  • A summary block with overall percentage, final letter grade, and PASS/FAIL status.
  • Signature lines for the Class Teacher, Examination Officer and Principal.
  • A footer noting the document is system-generated.
