1. Overview & VTU Simulation

This application is designed to simulate the result management workflow of Visvesvaraya Technological University (VTU), Belagavi. It mirrors the way a VTU affiliated college and the Office of the Registrar (Evaluation) might publish provisional results to students after a Semester End Examination.

Although the data is stored locally in a SQLite database file (rather than the live VTU examination system), the look-and-feel, identifiers, grading scheme, and PDF marks card layout are all modelled on VTU conventions:

  • Students are identified by their USN (University Seat Number).
  • Marks are graded according to the VTU CBCS 2022 Scheme letter-grade table.
  • The downloadable PDF is titled "Official Provisional Result Sheet", in line with VTU phrasing.
  • Each record carries the student's college / institute name, since VTU is an affiliating university.

1.1 User Flows

  • Public lookup — anyone can visit /results, identify themselves with USN + name and download their provisional result.
  • Admin management — an authenticated administrator can add, edit and delete student records from /admin.

2. Technologies Used

Python 3.11

Modern Python runtime — chosen for readability and a rich ecosystem of web and PDF libraries.

Flask

Minimal Python web framework providing routing, sessions, and Jinja2 templating without the overhead of a full-stack framework.

Jinja2 Templates

HTML is rendered server-side, with a shared base.html layout extended by every page.

SQLite

File-based relational database accessed through Python's built-in sqlite3 module. All records persist to a single database.db file alongside the app, so the project can be hosted on any free platform with no external database setup.

ReportLab

Pure-Python PDF generation library used to produce the official-looking VTU-style provisional result sheet.

Vanilla HTML & CSS

No frontend framework — a single style.css defines the design tokens and components.

4. Data Model

Records live in a single SQLite table called students, created automatically on first run inside database.db: 

CREATE TABLE students (
    id      TEXT PRIMARY KEY,   -- USN, normalized to uppercase
    name    TEXT NOT NULL,
    college TEXT NOT NULL DEFAULT '',
    marks   TEXT NOT NULL       -- JSON blob: { "Mathematics": 92, ... }
);

The application loads each row into a Python dictionary that looks like this:

{
  "id":      "1VT22CS001",
  "name":    "Jane Doe",
  "college": "RV College of Engineering, Bengaluru",
  "marks": {
    "Mathematics": 92,
    "Science":     88,
    "English":     76,
    "History":     81,
    "Computer":    95
  }
}

The USN is the primary identifier — it is normalized to uppercase, used as the SQLite primary key, displayed prominently on every screen, and printed on the PDF marks card. The college / institute field is required so that the marks card can show the student's affiliated institution alongside the university name. Marks are stored as a JSON string in the marks column so that the list of subjects can evolve without requiring schema migrations.

Because everything lives in a single database.db file on disk, deploying to any free host (Render, Railway, Fly.io, PythonAnywhere, a personal VPS, etc.) is just a matter of running the app — the file will be created on first start and persist across restarts on any host with a writable filesystem.

5. Grading System (VTU 2022 Scheme)

Whenever a result is displayed (on screen or in the PDF), totals and grades are computed from the stored marks. The system follows the VTU Choice Based Credit System (CBCS) — 2022 Scheme letter-grade mapping.

5.1 Letter Grades

For each subject and for the overall percentage, the letter grade is derived from the marks as follows:

Marks (out of 100)GradeDescription
90 – 100OOutstanding
80 – 89A+Excellent
70 – 79AVery Good
60 – 69B+Good
50 – 59BAbove Average
45 – 49CAverage
40 – 44PPass
Below 40FFail

5.2 Totals & Percentage

  • Total = sum of marks across all five subjects.
  • Maximum = number of subjects × 100 = 500.
  • Percentage = total ÷ number of subjects, rounded to two decimal places.
  • Final Grade = letter grade for the percentage, using the VTU table above.

5.3 Pass / Fail Status

Following VTU's per-subject minimum, a student is marked PASS only if they score at least 40 out of 100 in every subject. A score below 40 in any subject results in an overall FAIL, regardless of the percentage average.

5.4 Input Validation

  • USN, Name and College are required and trimmed of whitespace; USNs are normalized to uppercase.
  • Marks for every subject are required and must be whole numbers between 0 and 100.
  • Adding a record with an existing USN is rejected; editing a record to use a USN that already belongs to another student is also rejected.

6. Security Features

6.1 Admin Authentication

The /admin dashboard and all add / edit / delete routes are protected by a @login_required decorator. Credentials are validated against the ADMIN_USERNAME and ADMIN_PASSWORD environment variables (defaulting to admin / admin123 for first-run convenience).

6.2 Server-Side Sessions

Flask's signed-cookie sessions track the logged-in admin and remember the student who just successfully looked up a result (so they — and only they — can download the matching PDF). The signing key comes from SESSION_SECRET.

6.3 Two-Factor Result Lookup

On the public results page, a student must enter both their USN and their full registered name. Even if someone knows or guesses a USN, they cannot view marks without also knowing the name on file. Error messages are intentionally generic so the form does not reveal whether a given USN exists.

6.4 Math Captcha

Each visit to the results page generates a fresh arithmetic challenge stored in the session. The form only accepts a lookup when the captcha answer matches, and each captcha is consumed and regenerated on submit so it cannot be replayed. This blocks naïve bots from scraping results by enumerating USNs.

6.5 PDF Download Authorization

The PDF route does not accept a USN parameter. Instead it reads the USN and name from a session token issued only after a successful captcha-verified lookup, then re-validates the stored record. This prevents direct-link scraping of arbitrary students' PDFs.

6.6 Other Hardening

  • Destructive admin actions only respond to POST, with a confirmation prompt.
  • All user input is escaped automatically by Jinja2's default autoescaping.
  • The SQLite database is accessed only through parameterized queries (? placeholders), preventing SQL injection.

7. PDF Marks Card

After a successful lookup, the student can click Download as PDF to receive a VTU-style Official Provisional Result Sheet, built on demand with ReportLab. The layout includes:

  • University header — name, address, and the Office of the Registrar (Evaluation).
  • The page sub-title "OFFICIAL PROVISIONAL RESULT SHEET".
  • Student details: USN, name, college / institute, scheme (CBCS — 2022 Scheme), date of issue, and examination label.
  • A numbered subject-wise table showing maximum marks, marks obtained, per-subject grade and pass/fail.
  • A summary block with overall percentage, final grade and PASS/FAIL.
  • Signature lines for the Class Teacher, Examination Officer and Registrar (Evaluation).
  • A footer noting the document is system-generated.

8. Institutional Branding

The application is shipped pre-branded for Visvesvaraya Technological University, but the branding is intentionally not hard-coded throughout the codebase. Three environment variables control how the institution appears on the PDF marks card and other VTU references:

VariableDefaultWhere it appears
INSTITUTION_NAME Visvesvaraya Technological University Top heading of the PDF marks card and the footer disclaimer.
INSTITUTION_TAGLINE Jnana Sangama, Belagavi - 590018, Karnataka Address line directly under the university name on the PDF.
INSTITUTION_OFFICE Office of the Registrar (Evaluation) Issuing-office line on the PDF, mirroring VTU's examination wing.

8.1 How University-specific PDFs are Generated

  1. When a student successfully looks up their result, a session token (USN + name) is issued.
  2. Clicking Download as PDF calls /results/pdf, which re-validates that token against the stored record.
  3. The handler invokes build_marks_card_pdf(), which uses ReportLab's flowables (Paragraph, Table, HRFlowable) to lay out the document on A4 paper.
  4. The header pulls INSTITUTION_NAME, INSTITUTION_TAGLINE, and INSTITUTION_OFFICE from the environment, so the same code can produce a marks card for any university simply by changing those values.
  5. The body table is populated from the student's stored record, with letter grades computed via grade_for_marks() using the VTU 2022 Scheme.
  6. The completed PDF is streamed back to the browser as marks_card_<USN>.pdf.

Because the per-student data carries its own college field, a single deployment can issue branded VTU result sheets for students from many different affiliated colleges without any code changes.
